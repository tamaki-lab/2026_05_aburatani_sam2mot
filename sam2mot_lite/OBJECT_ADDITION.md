# SAM2.1における「途中入力」と「動的オブジェクト追加」の技術設計書

本ドキュメントでは、モデルアーキテクチャの制約を突破し、SAM2.1において時間的メモリを保持したまま「動画の途中から新規オブジェクトを追加する（Milestone 5）」ための技術設計と、CPUオフロード有効化に伴うデバイス競合の解決策について解説します。

---

## 1. SAM2.1のネイティブ「途中入力」と「新規オブジェクト追加」の違い

SAM2.1では公式にインタラクティブな途中入力がサポートされましたが、我々がマルチオブジェクトトラッキング（MOT）で必要とする要件とは決定的な違いがあります。

| 機能 | SAM2.1のネイティブ途中入力 | SAM2MOT-liteの動的オブジェクト追加 |
| :--- | :--- | :--- |
| **対象** | **既存のオブジェクト**に対するクリックやボックスの追加 | **全く新しいオブジェクト**の途中出現による追加 |
| **オブジェクト数（Batch数）**| 動画の最初から最後まで**不変**（例: 3個で固定） | 動画の途中で動的に**増加**（例: 3個 $\rightarrow$ 4個） |
| **動作フレーム** | 既存オブジェクト의 追跡のズレを中間の任意のフレームで補正する | 新しく画面内に登場したオブジェクトの追跡を開始する |
| **アーキテクチャ上の制約**| なし（モデル定義内のバッチサイズに変更がないため） | **大（過去フレームとのBatch次元不整合によるクラッシュ）** |

---

## 2. 新規オブジェクト追加時の技術的課題

SAM2.1は内部のメモリバンクに過去のオブジェクトの見た目や位置の記憶（`pred_masks`, `obj_ptr`, `maskmem_features`など）を蓄積しています。
動画の途中でオブジェクトを増やす（バッチサイズを $B$ から $B+1$ に拡張する）と、以下の不整合が発生します。

```mermaid
graph TD
    subgraph 過去フレーム (t=1 ~ 12)
        A["既存オブジェクト (3個)"]
        style A fill:#f9f,stroke:#333,stroke-width:2px
    end
    subgraph 現在フレーム (t=13)
        B["既存オブジェクト (3個) + 新規オブジェクト (1個)"]
        style B fill:#bbf,stroke:#333,stroke-width:2px
    end
    A -->|"アテンション計算でバッチサイズ不整合が発生"| B
    B -->|"RuntimeError: Sizes of tensors must match!"| C[クラッシュ]
    style C fill:#f66,stroke:#333,stroke-width:2px
```

### 💡 解決策：過去メモリスロットの動的拡張（Dynamic Batch Padding）
SAM2.1のメモリバンクを遡り、過去のすべてのフレームデータに対し、**新規オブジェクト用の空のスロットをダミーデータで埋める（パディングする）**ことで、モデル内部のバッチサイズを過去に遡って整合させます。

---

## 3. アプローチの比較

動画途中のオブジェクト数変更に対して、考えられる3つの設計アプローチのトレードオフです。

| アプローチ | メリット | デメリット | 採用判定 |
| :--- | :--- | :--- | :--- |
| **1. 動的パディング（本実装）** | 既存オブジェクトの過去の**時間的メモリを100%保持**できるため、遮蔽や交差に極めて強い。 | SAM2の非公開な内部ステート（`inference_state`）を解読して操作する必要があり、難易度が高い。 | **採用（最強の性能）** |
| **2. ステートリセット方式** | 実装が最も容易。新オブジェクト登場時に`init_state`を呼び出し、全員を再プロンプトする。 | **過去の記憶が完全に消失する**ため、DanceTrackなどの激しい交差環境下でIDスイッチが多発する。 | 非推奨 |
| **3. トラッカー複数並列実行** | オブジェクトごとに完全に独立した predictor を走らせるため、バッチサイズ問題が起きない。 | オブジェクト数に比例して**VRAM消費量が破滅的に増加**し、実行速度も極端に低下する。 | 非推奨（VRAM OOM） |

---

## 4. SAM2.1「CPUオフロード」有効化によるデバイス競合と解決策

### ⚠️ エラーの発生条件
SAM2.1の超メモリ節約モードである **CPUオフロード** (`offload_state_to_cpu=True`) を有効にすると、SAM2.1はVRAMを空けるため、アテンション計算に使われていない過去フレームのメモリエンコーディングを**自動的にGPUからCPUへと退避**させます。

この状態で、過去フレームのテンソルを拡張（`torch.cat`）しようとした際、元のパディングコードは固定のデバイス（例: `cuda:0`）で拡張テンソルを作ってしまったため、以下の競合が発生しました。

> **[RuntimeError]** Expected all tensors to be on the same device, but got tensors is on cpu, different from other tensors on cuda:0

```
[CPUに退避された過去テンソル] (CPU)   <--- 結合(torch.cat)しようとして衝突！
           ▲
[新規にGPU上に作られたダミーパディング] (GPU/cuda:0) 
```

### 🛠️ 動的デバイス検知による解決（Dynamic Device Awareness）
この問題を解決するため、パディング作成時に「グローバルなデバイス設定」を使うのではなく、**拡張対象となる個々のテンソル自身のデバイス属性（`.device`）を動的に読み取り、同じデバイス上にパディングを作成する**ように修正しました。

```python
# 各テンソル自身のデバイスを動的に取得してパディングを作成する
dev_pos = pos.device # CPUに退避されていれば CPU、GPUにあれば GPU が取得される
pad_pos = torch.zeros((pad_size, C_pos, H_pos, W_pos), device=dev_pos)
padded_pos_enc.append(torch.cat([pos, pad_pos], dim=0))
```

これにより、**「SAM2.1の超メモリ節約（CPUオフロード）」と「時間的メモリを失わない動的バッチ拡張」が完全な安全性をもって共存可能**になりました。
